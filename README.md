# krotVPN 🦫

Собственный VPN-протокол в стиле VLESS с сервером и клиентом в **TUN-режиме**,
спроектированный так, чтобы его трафик был **неотличим от обычного HTTPS** и
устойчив к анализу трафика и фингерпринтингу.

Крот, который роет туннели.

> Учебно-исследовательский проект про приватность и сетевые транспорты.
> Используйте на инфраструктуре, которой владеете, и в рамках вашего
> законодательства.

## Быстрый старт

Одна команда на Linux-хосте (amd64/arm64): скачивает последний релиз, ставит
бинарники в `/usr/local/bin`, кладёт копию `krotctl` в текущую папку и запускает
интерактивный мастер настройки:

```bash
curl -fsSL https://raw.githubusercontent.com/Filaind/krotVPN/main/install.sh | sudo bash
```

В мастере просто жми Enter, чтобы принять значения по умолчанию (под каждым
вопросом — подсказка, что это и что рекомендуется). На сервере получишь
креды клиента, которые надо ввести в мастере на клиентской машине.

Дальше можно управлять сервисом локально установленной копией:

```bash
sudo ./krotctl server status     # статус сервиса / TUN / NAT
sudo ./krotctl uninstall         # полное удаление krot с хоста
```

Переменные окружения для установщика: `KROT_VERSION=v1.0.0` — закрепить версию,
`KROT_BINDIR` — каталог системной установки, `KROT_LOCAL_DIR` — куда положить
`./krotctl`, `KROT_NO_RUN=1` — только установить, не запускать мастер.
Подробнее — в разделе [Установка](#установка).

## Принцип маскировки

Многие обфусцированные транспорты выдаёт паттерн **TLS-in-TLS**: сразу после
внешнего TLS-рукопожатия идёт второе, TLS-образное рукопожатие (ClientHello
≈ 517 Б, затем всплеск сертификата 2–4 КБ) — характерная сигнатура, которую видно
по размерам записей.

krotVPN убирает признаки, по которым туннель отличают от обычного веб-трафика:

| Сигнал для анализатора | Контрмера krot |
|------------------------|----------------|
| **Отпечаток ClientHello (JA3/JA4)** | Клиент через **uTLS** шлёт настоящий **Chrome** ClientHello. |
| **Инспекция SNI / сертификата** | Настоящий **TLS 1.3** на ваш домен. В TLS 1.3 сертификат *зашифрован*, поэтому пассивный наблюдатель видит лишь `SNI=ваш-домен` + Chrome-отпечаток — как у обычного сайта. |
| **Распознаваемое «второе рукопожатие»** | Его **нет**. Обмен ключами X25519 + PSK-аутентификация вшиты в обычный **WebSocket-апгрейд** (в куках/заголовках). После единственного настоящего TLS на проводе — только непрозрачные WS-кадры. |
| **TLS-in-TLS *вашего* трафика** | Слой кадров добивает **первые кадры каждого потока почти до размера MTU** (в стиле XTLS-Vision), стирая размерный всплеск вложенного рукопожатия; дальше — лёгкий случайный паддинг. |
| **Активное зондирование** | Любой запрос без валидной аутентификации (неверный PSK, путь, голый `GET /`) **реверс-проксируется на реальный decoy-сайт**. Зонд видит обычный сайт с валидным сертификатом. |
| **Replay / capture-replay** | Токен аутентификации связывает эфемерный ключ + метку времени + статический ключ сервера; проверяется окном свежести и скользящим **replay-фильтром**. |

Крипта: **X25519** (эфемерный-эфемерный для forward secrecy *и* эфемерный-статический,
чтобы клиент аутентифицировал сервер) → **HKDF-SHA256** → **ChaCha20-Poly1305** со
счётчиком-nonce. Аутентификация — **PSK** (ваш «UUID», как в VLESS).

## Возможности

- **Канальный бондинг.** Один логический туннель = N параллельных WebSocket-каналов.
  Пакеты раскидываются по каналам через **flow-hash** (каждое соединение закреплено
  за одним каналом → нет переупорядочивания → внутренний TCP не режет окно). Пробивает
  per-flow троттлинг провайдера: при параллельной нагрузке скорость ~×N.
- **SOCKS-режим.** Клиент поднимает локальный SOCKS5, чей исходящий трафик уходит в
  туннель через source-routing. Удобно для чейнинга (xray → krot → EU).
- **Три режима маршрутизации клиента**: `full` (весь трафик в туннель), `socks`
  (SOCKS5-прокси), `manual` (только TUN, маршруты задаёт оператор).
- **`krotctl`** — единый CLI для развёртывания (с интерактивным мастером).
- **BBR** на серверах + увеличенные TCP-буферы для скорости.
- **Decoy-fallback** и **replay-фильтр** против активного зондирования.

## Архитектура

```
           хост клиента                                       хост сервера (Linux)
   ┌─────────────────────────────────┐              ┌──────────────────────────────────┐
   │ приложения → маршрут → krot0     │              │  krot0 (TUN) ──► NAT/MASQUERADE   │──► интернет
   │            (TUN)                 │              │        ▲                          │
   │             │ IP-пакеты          │              │        │ IP-пакеты                │
   │        krot-client               │  TLS 1.3     │   krot-server                     │
   │  uTLS Chrome → WS → AEAD-кадры   │◄═(uTLS Chrome)═►│  WS-апгрейд(auth) → AEAD-кадры │
   │     × N бондинг-каналов          │  выглядит как │      decoy при неверном auth      │
   └─────────────────────────────────┘  обычный HTTPS└──────────────────────────────────┘
```

Сервер выдаёт каждому клиенту адрес из `tun_cidr` и маршрутизирует ответы по
адресу назначения. Поддерживается несколько клиентов.

### Цепочка с xray (опционально)

```
[устройство] → xray (любой inbound) → SOCKS → krot-client (бондинг) → krot-server → интернет
```
xray принимает подключения устройств и отдаёт исходящий трафик в krot, который
выступает замаскированным upstream-каналом. krot-client в этой схеме работает в
`socks`-режиме.

## Установка

Самый простой путь — установщик одной командой (см. [Быстрый старт](#быстрый-старт)
в начале README). Ниже — ручные варианты.

### Готовые бинарники (релизы)

Каждый git-тег `v*` автоматически собирается через GitHub Actions в архивы под
Linux **amd64** и **arm64** (в каждом — `krot-server`, `krot-client`,
`krot-keygen`, `krotctl` + примеры конфигов и systemd-юниты). Скачать со страницы
**Releases**:

```bash
VER=v0.1.0; ARCH=amd64          # или arm64
curl -fsSL -O https://github.com/krot-vpn/krot/releases/download/$VER/krot-$VER-linux-$ARCH.tar.gz
curl -fsSL -O https://github.com/krot-vpn/krot/releases/download/$VER/SHA256SUMS.txt
sha256sum -c SHA256SUMS.txt --ignore-missing
tar xzf krot-$VER-linux-$ARCH.tar.gz
sudo install krot-$VER-linux-$ARCH/krot* /usr/local/bin/
krotctl version
```

### Сборка из исходников

Нужен Go 1.22+.

```bash
make all            # бинарники под хост -> ./bin
make dist           # статические Linux amd64 + arm64 -> ./dist
make test           # тесты рукопожатия/кадров/бондинга (в т.ч. -race)
```

Сервер и клиент работают **только на Linux** (TUN/маршруты/NAT). На macOS
`make all` соберёт полезным только `krot-keygen` и `krotctl` (для генерации/рендера).

### CI / релизы

- `.github/workflows/ci.yml` — на каждый push/PR в `main`: проверка `gofmt`,
  `go vet`, `go test -race`, кросс-сборка под amd64/arm64.
- `.github/workflows/release.yml` — на тег `v*`: собирает архивы, считает
  `SHA256SUMS.txt` и публикует GitHub Release. Версия прошивается в бинарь
  (`krotctl version`).

```bash
git tag v0.1.0 && git push origin v0.1.0   # запускает сборку релиза
```

## Быстрый старт через `krotctl` (рекомендуется)

`krotctl` — это **on-node** инструмент: копируете один бинарь на машину и
запускаете локально. Он генерирует секреты, рендерит конфиг, ставит systemd-юнит
и стартует сервис. Заменяет ручную возню с конфигами и юнитами.

### Интерактивный мастер

Проще всего — запустить без аргументов:

```bash
sudo krotctl
```

Мастер спросит роль (server/client) и по шагам соберёт параметры (с дефолтами в
скобках — Enter принимает дефолт), покажет **эквивалентную команду** и спросит
подтверждение перед применением.

### Сервер (EU)

```bash
# на сервере, одной командой: ключи + конфиг + systemd + старт
sudo krotctl server up -sni en.wikipedia.org -decoy https://en.wikipedia.org

# с реальным сертификатом Let's Encrypt вместо self-signed:
sudo krotctl server up -sni your-domain.example \
     -cert /etc/letsencrypt/live/your-domain.example/fullchain.pem \
     -key  /etc/letsencrypt/live/your-domain.example/privkey.pem \
     -decoy https://your-domain.example
```
После старта команда печатает **клиентские креды** (`sni`, `path`, `psk`,
`server_pub`) — их переносите на клиент.

Прочие команды:
```bash
sudo krotctl server status     # сервис, TUN, NAT, листенер
sudo krotctl server down        # стоп + откат NAT/TUN
```

### Клиент (Linux)

```bash
# режим full — весь трафик хоста через туннель
sudo krotctl client up -mode full \
     -server 46.224.77.233 -sni en.wikipedia.org -path /api/v2/xxx \
     -psk <BASE64> -server-pub <BASE64>

# режим socks — поднять SOCKS5 для чейнинга (xray и т.п.)
sudo krotctl client up -mode socks -name xray \
     -server 46.224.77.233 -sni en.wikipedia.org -path /api/v2/xxx \
     -psk <BASE64> -server-pub <BASE64> -socks 127.0.0.1:1080

sudo krotctl client status -name xray
sudo krotctl client down  -name xray
```

> **Несколько клиентов на одной машине.** Каждый инстанс задаётся флагом `-name`:
> свой конфиг `client-<name>.json`, свой systemd-юнит `krot-client@<name>`, своё
> TUN-устройство и (в socks-режиме) своя таблица маршрутизации. Так на одном шлюзе
> мирно живут, например, WG-uplink и xray-uplink.

#### Флаги `client up`
| флаг | назначение |
|------|------------|
| `-name` | имя инстанса (по умолчанию `default`) |
| `-mode` | `full` \| `socks` \| `manual` |
| `-server` / `-port` | адрес и порт сервера |
| `-sni` / `-path` | TLS-имя на проводе + секретный путь |
| `-psk` / `-server-pub` | секреты сервера (base64) |
| `-channels` | число бондинг-каналов (по умолчанию 4) |
| `-mtu` / `-dns` / `-insecure` | MTU / DNS / пропуск проверки серта |
| `-socks` | (socks) адрес SOCKS5, напр. `127.0.0.1:1080` |
| `-socks-table` | (socks) таблица policy-routing (по умолчанию выводится из имени) |

## Ручное развёртывание (без `krotctl`)

### 1. Генерация секретов
```bash
./bin/krot-keygen
```
Печатает **PSK**, приватный/публичный ключи сервера, случайный **секретный путь** и
готовые шаблоны `server.json` / `client.json`. `psk` и `server_priv` держите в секрете.

### 2. Сервер
```bash
sudo mkdir -p /etc/krot && sudo cp server.json /etc/krot/
sudo cp dist/linux-amd64/krot-server /usr/local/bin/
sudo cp deploy/krot-server.service /etc/systemd/system/
sudo systemctl enable --now krot-server
journalctl -u krot-server -f
```
Сервер создаёт `krot0`, включает форвардинг и ставит MASQUERADE + FORWARD
(WAN-интерфейс определяется автоматически; переопределяется `wan_iface`). При
остановке откатывает NAT и удаляет устройство.

> Порт 443 должен быть свободен. Если нужен и реальный веб-сервер — поднимите его
> на другом порту и укажите его в `decoy` (напр. `"decoy": "http://127.0.0.1:8080"`).

### 3. Клиент (Linux)
`server_ip` = IP сервера, `sni` = ваш домен.
```bash
sudo mkdir -p /etc/krot && sudo cp client.json /etc/krot/
sudo cp dist/linux-amd64/krot-client /usr/local/bin/
sudo krot-client -config /etc/krot/client.json
```
Дозванивается по **IP**, но шлёт `SNI=ваш-домен`, поднимает `krot0`, прибивает
host-route к серверу через реальный шлюз и заворачивает default в туннель.
**Ctrl-C аккуратно восстанавливает маршруты и DNS.**

Проверка:
```bash
curl https://api.ipify.org      # должен показать IP сервера
```

## Справочник конфигов

**server.json**

| ключ | смысл |
|------|-------|
| `listen` | адрес привязки, по умолчанию `:443` |
| `sni` | TLS-имя (для self-signed серта) |
| `path` | секретный WebSocket-путь; всё прочее → decoy |
| `psk` | base64 32-байтного PSK |
| `server_priv` | base64 32-байтного статического X25519 |
| `cert_file` / `key_file` | PEM цепочка + ключ (Let's Encrypt) |
| `self_signed` | только для тестов; генерит ECDSA P-256 |
| `decoy` | URL, куда реверс-проксируется неаутентифицированный трафик |
| `tun_name` / `tun_cidr` | TUN-устройство + подсеть-шлюз (`10.8.0.1/24`) |
| `dns` | DNS, выдаваемый клиентам |
| `wan_iface` | интерфейс egress для NAT (`""` = авто) |
| `mtu` | MTU туннеля (по умолчанию 1320) |

**client.json**

| ключ | смысл |
|------|-------|
| `server_ip` / `server_port` | что реально дозванивается |
| `sni` | TLS-имя на проводе (ваш домен) |
| `path` / `psk` / `server_pub` | должны совпадать с сервером |
| `insecure` | пропуск проверки серта (только с `self_signed`) |
| `pin_sha256` | опц. base64 SHA-256 SPKI серта сервера |
| `tun_name` / `mtu` | устройство / MTU (MTU должен совпадать с сервером) |
| `channels` | число бондинг-каналов (по умолчанию 4) |
| `dns` | опц. переопределение DNS |
| `route_mode` | `full` (по умолч.) \| `manual` (только TUN, без перехвата) |
| `socks_listen` | если задан — поднять SOCKS5 (напр. `127.0.0.1:1080`) |
| `socks_table` | таблица policy-routing для socks-режима |

## Настройка маскировки

Паддинг живёт в `internal/frame` (`DefaultPad`): первые `BurstRecords` кадров
добиваются до `[BurstMin,BurstMax]` байт, дальше `SteadyMax` случайных байт на
кадр. Больше burst-кадров / размеры = сильнее против TLS-in-TLS-анализа, но
больше накладных. Keepalive каждые ~20 с заодно служит cover-трафиком.

Бондинг настраивается флагом `-channels` (клиент) / полем `channels`. Эффект
умножения скорости проявляется при **параллельной** нагрузке (браузер, торренты,
несколько приложений), не на одиночной закачке.

Идеи для усиления (пока не сделано): timing-jitter, фронтинг WebSocket за CDN
(тогда SNI — это SNI CDN), HTTP/2-транспорт, UDP-транспорт (против TCP-over-TCP),
адаптивный паддинг под детект вложенного рукопожатия.

## Модель угроз и ограничения

- **Пассивный DPI:** видит `SNI=ваш-домен` + Chrome-отпечаток + TLS 1.3. Ни
  протокольного рукопожатия, ни plaintext-маркеров.
- **Активный зонд:** получает decoy-сайт по валидному сертификату.
- **Где всё ещё можно сгореть:** маскируется протокол, но не сам факт соединения с
  конкретным *доменом/IP*. Если эндпоинт известен наблюдателю — его можно
  заблокировать по IP, и от этого не спасает ни один протокол. Ротируйте
  эндпоинты, рассмотрите CDN-фронтинг.
- **Надёжность:** туннель не рвётся под потерями до ~20% (проверено netem), нет
  рассинхрона AEAD; но восстановление после полного разрыва держится на внешнем
  супервизоре (systemd `Restart=always`) — авто-redial отдельного канала пока не
  реализован.
- **Скорость:** транспорт TCP-over-TCP — на потерях скорость деградирует
  нелинейно; структурно лечится UDP-транспортом (в планах).
- Туннелируется только IPv4 (внутренние IPv6-пакеты отбрасываются).
- TUN/маршруты/NAT — только Linux. Клиенты под macOS/Windows — на будущее.
- Это research-grade код: примитивы стандартные, но конструкция формального аудита
  не проходила.
```
